# /document 워크플로우

기술 문서를 작성하는 프로세스입니다.

## 워크플로우 단계

### 1. 문서 유형 확인

작성할 문서 유형을 결정합니다.

| 유형 | 용도 | 대상 독자 |
|------|------|-----------|
| README | 프로젝트 소개 | 모든 사용자 |
| API 레퍼런스 | API 상세 정보 | 개발자 |
| 설치 가이드 | 설치 방법 | 사용자/개발자 |
| 튜토리얼 | 사용법 안내 | 초보 사용자 |
| 아키텍처 문서 | 시스템 구조 | 개발자 |
| CHANGELOG | 변경 이력 | 모든 사용자 |

### 2. 대상 분석

문서의 대상 독자를 파악합니다.

```
- 기술 수준
- 사전 지식
- 문서 사용 목적
- 필요한 정보 수준
```

### 3. 자료 수집

문서에 필요한 정보를 수집합니다.

```
- 코드베이스 분석 (code-analyzer 스킬 활용)
- 기존 문서 검토
- API 시그니처 확인
- 예제 코드 수집
```

### 4. 구조 설계

문서의 outline을 작성합니다.

```markdown
## README 구조 예시
1. 프로젝트명 + 한 줄 설명
2. 특징/목적
3. 빠른 시작
4. 설치
5. 사용법
6. 설정
7. API (해당시)
8. 기여 방법
9. 라이선스
```

### 5. 내용 작성

각 섹션을 작성합니다.

#### 작성 원칙
- **명확성**: 간결하고 이해하기 쉽게
- **완전성**: 필수 정보 누락 없이
- **예시 중심**: 코드 예시 적극 활용
- **일관성**: 용어와 형식 통일

#### 마크다운 스타일
```markdown
# 제목 (h1은 문서당 하나)
## 섹션 (h2)
### 하위 섹션 (h3)

- 목록은 하이픈 사용
- 짧고 명확하게

**강조** 또는 *이탤릭*

`인라인 코드` 또는

\`\`\`language
코드 블록
\`\`\`

> 인용 또는 주의사항

| 헤더 | 헤더 |
|------|------|
| 내용 | 내용 |
```

### 6. 예제 추가

실용적인 예제를 포함합니다.

```
- 기본 사용법 예제
- 고급 사용법 예제
- 일반적인 use case
- 에러 처리 예제
```

### 7. 검토 및 완성

문서 품질을 확인합니다.

#### 체크리스트
- [ ] 목적이 명확한가?
- [ ] 대상 독자에게 적합한가?
- [ ] 필수 정보가 모두 포함되었는가?
- [ ] 예제가 동작하는가?
- [ ] 문법/오타 오류가 없는가?
- [ ] 링크가 올바른가?
- [ ] 일관된 형식인가?

### API 문서
```markdown

## 개요

API 소개...

인증 방법...

## Endpoints

### POST /api/resource

**설명**: 리소스를 생성합니다.

**Request Body**
| 필드 | 타입 | 필수 | 설명 |
|------|------|------|------|
| name | string | ✅ | 이름 |

**Response**
\`\`\`json
{
  "id": "123",
  "name": "example"
}
\`\`\`

**에러 코드**
| 코드 | 설명 |
|------|------|
| 400 | 잘못된 요청 |
| 401 | 인증 필요 |
```
